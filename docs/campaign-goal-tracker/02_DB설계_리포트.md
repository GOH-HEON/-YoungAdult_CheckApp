# 목표대비 달성 — DB 설계 리포트

> 작성일: 2026-07-12 · DB: Supabase PostgreSQL · 기존 스키마: `supabase/schema.sql`
> 확정 결정: 단일 캠페인 · 전도/권유 전체 단일 카운터 · 목표 지표별 전체 1개
> 기존 자산 재사용: `public.members`(활성 194), `public.departments`(5), `public.users`

---

## 1. 설계 원칙

1. **기존 명단 재사용**: 155명은 이미 `members`에 존재(전원 일치 확인). 신규 인물 테이블 없음.
2. **최소 신규 테이블 3개**: `campaigns`, `campaign_participants`, `campaign_counter_logs`.
3. **카운터 = 로그 합계**: 전도/권유는 단일 정수 대신 delta 로그 → 되돌리기·감사·추이 지원.
4. **RLS 일관성**: 기존 `is_read_user()`(열람) / `is_admin_user()`(쓰기) 헬퍼 재사용.
5. **idempotent 마이그레이션**: `if not exists` / `begin…commit` 로 재실행 안전.

---

## 2. ERD (텍스트)

```
departments (기존) ──< members (기존) ──< campaign_participants >── campaigns
                                                                       │
                                          campaign_counter_logs >──────┘
users (기존) ──< (updated_by / created_by on 신규 3테이블)
```

- `campaigns 1 : N campaign_participants` (캠페인당 155행)
- `campaigns 1 : N campaign_counter_logs` (metric='전도'|'권유' delta 누적)
- `members 1 : N campaign_participants` (member는 캠페인마다 최대 1행 → unique)

## 3. 테이블 정의

### 3.1 `campaigns` — 캠페인 + 지표별 전체 목표
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid pk | |
| name | text not null | 예: '2026 하계 전도캠페인' |

| goal_registration | int not null default 0 | 접수 목표 (기본 155 권장) |

| goal_evangelism | int not null default 0 | 전도 목표 |
| goal_invitation | int not null default 0 | 권유 목표 |
| is_active | boolean not null default true | |
| created_by | uuid → users(id) | |
| created_at / updated_at | timestamptz | |

### 3.2 `campaign_participants` — 접수/참여 목록 (분야 1)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid pk | |
| campaign_id | uuid not null → campaigns(id) on delete cascade | |
| member_id | uuid not null → members(id) | |

| — | unique(campaign_id, member_id) | 캠페인당 1인 1행 |
| — | check (not participated or registered) | 참여는 접수 전제 |

### 3.3 `campaign_counter_logs` — 전도/권유 카운터 (분야 2·3)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid pk | |
| campaign_id | uuid not null → campaigns(id) on delete cascade | |
| metric | text not null check (metric in ('전도','권유')) | |
| delta | int not null check (delta <> 0) | 보통 +1, 되돌리기 −1 |
| note | text | 전도 대상 메모(선택) |
| created_by | uuid → users(id) | 입력자(감사) |
| created_at | timestamptz not null default now() | |

> 달성 = `SUM(delta)`. 0 미만 방지는 앱 액션에서 처리(＋/− 시 현재 합계 확인).

## 4. 마이그레이션 DDL (`supabase/ops/08_campaign_goal_tracker.sql`)

```sql
-- 목표대비 달성: 캠페인 + 접수/참여 목록 + 전도/권유 카운터
-- Supabase SQL Editor에서 실행. 재실행 안전(idempotent).
begin;

-- 1) campaigns
create table if not exists public.campaigns (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  description text,
  start_date date,
  end_date date,
  goal_registration integer not null default 0 check (goal_registration >= 0),
  goal_participation integer not null default 0 check (goal_participation >= 0),
  goal_evangelism   integer not null default 0 check (goal_evangelism >= 0),
  goal_invitation   integer not null default 0 check (goal_invitation >= 0),
  is_active boolean not null default true,
  created_by uuid references public.users(id),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

-- 2) campaign_participants (접수/참여 목록)
create table if not exists public.campaign_participants (
  id uuid primary key default gen_random_uuid(),
  campaign_id uuid not null references public.campaigns(id) on delete cascade,
  member_id uuid not null references public.members(id),
  registered boolean not null default false,
  participated boolean not null default false,
  registered_at timestamptz,
  participated_at timestamptz,
  note text,
  updated_by uuid references public.users(id),
  updated_at timestamptz not null default now(),
  unique (campaign_id, member_id),
  constraint participated_requires_registered check (not participated or registered)
);
create index if not exists idx_camp_part_campaign on public.campaign_participants (campaign_id);

-- 3) campaign_counter_logs (전도/권유 단일 카운터)
create table if not exists public.campaign_counter_logs (
  id uuid primary key default gen_random_uuid(),
  campaign_id uuid not null references public.campaigns(id) on delete cascade,
  metric text not null check (metric in ('전도','권유')),
  delta integer not null check (delta <> 0),
  note text,
  created_by uuid references public.users(id),
  created_at timestamptz not null default now()
);
create index if not exists idx_camp_counter_lookup
  on public.campaign_counter_logs (campaign_id, metric);

-- updated_at 트리거(기존 set_updated_at 재사용)
drop trigger if exists trg_campaigns_updated on public.campaigns;
create trigger trg_campaigns_updated before update on public.campaigns
  for each row execute function public.set_updated_at();
drop trigger if exists trg_camp_part_updated on public.campaign_participants;
create trigger trg_camp_part_updated before update on public.campaign_participants
  for each row execute function public.set_updated_at();

commit;
```

## 5. RLS 정책 (`supabase/ops/08` 하단 또는 rls.sql 반영)

```sql
begin;
alter table public.campaigns             enable row level security;
alter table public.campaign_participants enable row level security;
alter table public.campaign_counter_logs enable row level security;

-- 열람: 로그인한 열람 권한 사용자 전원
drop policy if exists campaigns_read on public.campaigns;
create policy campaigns_read on public.campaigns
  for select using (public.is_read_user());
drop policy if exists camp_part_read on public.campaign_participants;
create policy camp_part_read on public.campaign_participants
  for select using (public.is_read_user());
drop policy if exists camp_counter_read on public.campaign_counter_logs;
create policy camp_counter_read on public.campaign_counter_logs
  for select using (public.is_read_user());

-- 쓰기: admin (staff 허용은 §6 결정 후 is_read_user 등으로 완화 가능)
drop policy if exists campaigns_write on public.campaigns;
create policy campaigns_write on public.campaigns
  for all using (public.is_admin_user()) with check (public.is_admin_user());
drop policy if exists camp_part_write on public.campaign_participants;
create policy camp_part_write on public.campaign_participants
  for all using (public.is_admin_user()) with check (public.is_admin_user());
drop policy if exists camp_counter_write on public.campaign_counter_logs;
create policy camp_counter_write on public.campaign_counter_logs
  for all using (public.is_admin_user()) with check (public.is_admin_user());
commit;
```

> 앱은 서버 액션에서 세션 클라이언트로 쓰기하므로 RLS가 실제 방어선이다. 서비스 롤(관리 클라이언트)로 우회 쓰기 시에도 액션 레벨 `requireAdminSession()` 재검증 유지.

## 6. 캠페인 생성 + 155명 시드

명단(155명)은 기존 `members`와 (이름 + 부서) 기준 100% 일치가 확인됐다. 캠페인 생성 후 해당 155명만 `campaign_participants`에 시드한다.

```sql
-- (a) 캠페인 1건 생성 (목표는 이후 UPDATE로 조정)
insert into public.campaigns (name, description, start_date, end_date,
  goal_registration, goal_participation, goal_evangelism, goal_invitation)
values ('2026 하계 전도캠페인', '청년회 여름 전도', '2026-07-01', '2026-08-31',
  155, 155, 45, 150)
returning id;  -- 이 id를 아래 :cid 로 사용
-- goal_registration=155, goal_participation=155, goal_evangelism=45, goal_invitation=150

-- (b) 155명 시드: 명단이 특정 부서 전원과 일치하지 않을 수 있으므로,
--     엑셀 (이름,부서) 목록을 임시 테이블로 올려 매칭하는 방식을 권장.
--     아래는 "부서 전체가 대상"인 경우의 예시(부서 5개 활성 멤버 전원).
insert into public.campaign_participants (campaign_id, member_id)
select :cid, m.id
from public.members m
where m.is_active = true
  and m.department_id in (1,2,3,4,5)
on conflict (campaign_id, member_id) do nothing;
```

- **정확 시드(권장)**: 엑셀의 (이름, 부서) 155행을 값 목록으로 만들어 `members`와 조인 후 insert.
  앱에서 처리하거나, 별도 시드 스크립트(`scripts/seed-campaign.ts`, 서비스 롤)로 1회 실행.
  → 개인정보(명단)를 리포지토리에 커밋하지 않도록 시드 데이터는 **로컬/비커밋**으로 다룬다.
- **[검증]** DB 대조 결과 155명 = 5개 부서 활성 멤버 중 일부. 단순 "부서 전원"과 개수가 다를 수 있으므로(활성 194 ≠ 155) 정확 시드 방식 사용 권장.

## 7. 앱 계층 매핑

| 화면 요소 | 쿼리 |
|-----------|------|
| 접수 달성 | `count(campaign_participants where campaign_id=? and registered)` |
| 참여 달성 | `count(campaign_participants where campaign_id=? and participated)` |
| 전도 달성 | `sum(delta) from campaign_counter_logs where campaign_id=? and metric='전도'` |
| 권유 달성 | `sum(delta) ... and metric='권유'` |
| 목록 표 | `campaign_participants join members join departments (campaign_id=?)` |
| 최근 기록 | `campaign_counter_logs where campaign_id=? and metric=? order by created_at desc limit N` |

TypeScript 타입은 `lib/` 하위에 `Campaign`, `CampaignParticipant`, `CounterLog` 추가.

## 8. 마이그레이션 실행 메모

- DDL(테이블/정책/함수)은 **DB 비밀번호 또는 Supabase 액세스 토큰**이 있어야 실행 가능.
  현재 `.env.local`에는 서비스 롤 키만 있어 DDL 직접 실행 불가 → SQL Editor 붙여넣기 또는 연결 문자열 제공 필요.
- 시드(INSERT)는 서비스 롤 키로 실행 가능(PostgREST/스크립트).
