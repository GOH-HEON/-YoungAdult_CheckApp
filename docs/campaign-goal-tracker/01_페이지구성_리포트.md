# 목표대비 달성 — 페이지 구성 리포트

> 작성일: 2026-07-12 · 대상 앱: YoungAdult_CheckApp (Next.js 16 App Router)
> 데이터 출처: `청년회_모임_2026-07-09_명단.xlsx` Sheet1(155명) + 기존 Supabase DB
> 확정 결정: 단일 캠페인 · 전도/권유는 전체 단일 카운터 · 목표는 지표별 전체 하나

---

## 1. 개요

전도캠페인 1건에 대해 **접수 / 참여 / 전도 / 권유** 4개 지표의 목표대비 달성 현황을 한 화면에서 관리한다.

| 지표 | 방식 | 단위 | 목표 |
|------|------|------|------|
| 접수 | **목록** (155명 개인별 체크) | 사람 | 전체 목표 1개 (기본 155) |
| 참여 | **목록** (155명 개인별 체크) | 사람 | 전체 목표 1개 (기본 155) |
| 전도 | **단일 카운터** (＋/− 누적) | 숫자 | 전체 목표 1개 |
| 권유 | **단일 카운터** (＋/− 누적) | 숫자 | 전체 목표 1개 |

- **[측정]** 명단 155명 = 보라부 39 · 뿌리부 39 · 소명부 35 · 은혜부 35 · 회장단 7 (형제 82 / 자매 73).
- **[측정]** 155명 전원이 기존 `members`(활성 194명)와 (이름+부서)로 100% 일치 → 신규 인원 입력 없이 기존 명단 재사용.

## 3. 화면 레이아웃 (상→하)

목업 이미지: `03_페이지_가안.png` 참조.

### 3.1 헤더
- 페이지 제목 "목표대비 달성" + 캠페인 선택 드롭다운(현재 활성 캠페인, 기간 표시).
- 캠페인이 여러 개가 되어도 확장 가능 (기본은 활성 1개).

### 3.2 요약 카드 4개 (접수 · 참여 · 전도 · 권유)
- 각 카드: 지표명 + 배지(목록/카운터) + `달성 / 목표` 큰 숫자 + 달성률(%) + 진행 바.
- 색상: 접수=파랑, 참여=초록, 전도=보라, 권유=주황 (앱 토큰 `--primary #2563eb` 계열 + emerald).
- 달성률 = round(달성 / 목표 × 100). 목표 0이면 "—" 표시(0 나눗셈 방지).

### 3.3 분야 1 — 접수 / 참여 (목록)
- **부서 필터 칩**: 전체 / 보라부 39 / 뿌리부 39 / 소명부 35 / 은혜부 35 / 회장단 7.
- **이름 검색** 입력.
- **표 컬럼**: 이름 · 성별(형제/자매 태그) · 소속부서 · 접수(토글) · 참여(토글).
  - 접수 토글: `미접수 ↔ 접수완료`, 참여 토글: `미참여 ↔ 참여완료`.
  - 토글 클릭 시 서버 액션 호출 → 낙관적 UI 업데이트.
  - 정책: **참여완료는 접수완료를 전제**로 한다(참여 ON 시 접수 자동 ON). 접수 OFF 시 참여도 OFF.
- **부서별 진행 카드**: 부서마다 참여 n/총원 + 진행 바 (부서 비교용, 목표 자체는 전체 1개지만 부서 현황은 참고 표시).

### 3.4 분야 2·3 — 전도 / 권유 (단일 카운터, 좌우 2단)
- 좌: 큰 달성 숫자 + `목표 N명` + 진행 바 + 달성률.
- 버튼: `−1` (되돌리기) / `＋ 전도(권유) 추가`. 클릭 시 delta 로그 1건 생성.
  - 추가 시 선택 입력: 메모(전도 대상자 이름 등, 선택), 증가 수(기본 1).
- 우: **최근 기록** 리스트 (delta · 입력자 · 시각). 감사/되돌리기 근거.
- 하단 안내: "＋/− 버튼으로 누적, 모든 변경은 기록에 남습니다".

## 4. 상호작용 / 서버 액션

| 액션 | 입력 | 처리 | 반환 |
|------|------|------|------|
| `toggleRegistration` | campaignId, memberId, next(boolean) | `campaign_participants` upsert (registered, registered_at) | 갱신 행 |
| `toggleParticipation` | campaignId, memberId, next(boolean) | participated 갱신, next=true면 registered도 보장 | 갱신 행 |
| `addCounter` | campaignId, metric('전도'\|'권유'), delta(int), note? | `campaign_counter_logs` insert | 신규 합계 |
| `updateGoals` | campaignId, {goal_registration, goal_participation, goal_evangelism, goal_invitation} | `campaigns` update (admin 전용) | 갱신 캠페인 |

- 모든 쓰기 액션은 `requireAdminSession()`(또는 지정 권한) 통과 후 실행, 서버에서 재검증.
- 달성 집계는 서버에서 계산(신뢰 경계): 접수/참여 = count, 전도/권유 = `sum(delta)`.

---

## 5. 집계 규칙 (서버 계산)

```
접수 달성 = COUNT(campaign_participants where campaign_id=? and registered)
참여 달성 = COUNT(campaign_participants where campaign_id=? and participated)
전도 달성 = COALESCE(SUM(delta) where campaign_id=? and metric='전도'), 0)
권유 달성 = COALESCE(SUM(delta) where campaign_id=? and metric='권유'), 0)
달성률(%) = 목표>0 ? ROUND(달성/목표*100, 1) : null
```

- 카운터를 로그 합계로 두는 이유: **되돌리기(−1) · 입력자 감사 · 시계열 추이**를 모두 지원. 단일 정수 컬럼 대비 안전.
- 성능: 로그가 커지면 `campaign_id, metric` 인덱스로 `SUM` 조회. 캠페인당 수천 건 수준이면 문제 없음 [추정].

## 6. 확인 필요 항목 (설계 확정 전 1차 검토)

1. **쓰기 권한 범위**: 접수/참여 토글·카운터 증가를 admin만? staff·chairboard도 허용? (현재 안: admin 쓰기 / 전 열람. 부서 간사가 입력해야 하면 staff 허용 필요.)
2. **접수/참여 대상 고정**: 155명을 캠페인 시작 시 스냅샷으로 고정할지, `members` 활성 변동을 실시간 반영할지. (안: 캠페인 생성 시 155명을 `campaign_participants`에 시딩 → 고정.)
3. **목표 기본값**: 접수/참여 목표를 155(전원)로 자동 설정? 전도/권유 목표 숫자는 사용자 입력 필요.
4. **−1 하한**: 카운터 합계가 0 미만으로 내려가지 않도록 막을지(안: 0 미만 방지).

> 위 4개는 값만 정해주시면 DB 시드/기본값에 반영합니다. 미정 시 괄호 안 "안"을 기본값으로 진행합니다.
